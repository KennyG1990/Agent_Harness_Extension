# Research-to-Implementation Gap Analysis

## Reconciled Current Status (2026-07-11)

## Phase 68 - Enforced Universal Task Workflow (implemented; lifecycle/bypass/host/visual gates PASS)

Reconciled gap: Forge already had a five-task agent loop, firewall stages, artifacts, reviewer gates, evidence, and terminal AAR, but the user's universal `CLASSIFY -> PLAN -> BASELINE -> RECONCILE -> DOCUMENT PLAN -> IMPLEMENT -> VALIDATE -> REVIEW -> DOCUMENT CLOSE -> AAR` workflow existed only as prose. Task status could be manually advanced without a durable baseline/acceptance contract, and no deterministic gate prevented an otherwise role-valid mutation from skipping reconciliation and planning.

Implemented result: `workflowGovernance.ts` now creates a required versioned state object for every run. Conservative deterministic lane classification selects Light only for documentation/text-only goals and defaults ambiguous or behavioral work to Full. Initialization captures a bounded baseline (workspace, version, Git revision/status, file count, prior Forge state, rollback method), compiles a structured acceptance contract, and marks classify/plan/baseline complete before model work. Full runs require accepted exploration before plan documentation; Light runs use the generated bounded one-line plan. Architect plans are deterministically augmented with Acceptance Contract, Validation, Negative Path, and Rollback sections when weak models omit them.

The workflow validator runs between role validation and the generic firewall. File/command implementation is rejected until baseline, reconcile, and documented plan are complete; evidence is rejected before validation; success is rejected until baseline, reconcile, plan, implementation/no-op proof, validation, review, and close are complete. Accepted tool events, automatic post-mutation oracles, no-change diff reviews, harness-authored green evidence, and terminal AAR advance only their corresponding stages. Workflow violations persist and enter the blocker/reflection system as `workflow_gate`; task-graph manipulation alone cannot bypass them. A successful run becomes `VERIFIED` only after AAR and all 11 stages complete.

Evidence persists in `.forge/workflow-governance.json` and human-readable `.forge/workflow-task-record.md`; both open in native editors. The required model prompt carries lane, current stage, stage statuses, acceptance criteria, and validation contract. Compact footer telemetry renders `flow stage/blocks` without another panel. Behavioral proof covers Full/Light/ambiguous classification, “fix README typo” Light classification, early mutation rejection with unchanged bytes, early evidence rejection, plan section completion, full stage lifecycle, blocker taxonomy, compatibility with layered firewall/network tests, no-op implementation proof, successful product rate split, and terminal `VERIFIED` with 11/11 completed stages. Inspected persisted fixture evidence records zero violations and all stage evidence; visual proof shows `flow review/0`.

Compile, smoke, 100/100 worker stress, extension-host E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.68.0`. The VSIX contains both compiled `workflowGovernance.js` and `UNIVERSAL_AI_TASK_WORKFLOW.md`. Honest boundary: the universal governance core is enforced, but project-specific adapter commands and live-domain gates still come from repository knowledge/configuration rather than a typed adapter registry; OS/socket containment and live eval gaps remain separate.

## Phase 67 - Verified Procedural Skill Memory (implemented; causal banking/retrieval/host/visual gates PASS, live uplift pending)

Reconciled gap: `HarnessState.skills` and `.forge/skill-registry.json` existed structurally but were always initialized to an empty array. AAR banked vague lessons from failures, yet Forge could not convert a verified successful recovery into an executable workflow or retrieve that workflow for a later related run. The research requirement for a skill registry was therefore presence-only.

Implemented result: pure deterministic `bankProceduralSkills` and `selectProceduralSkills` own the registry. Only terminal `success` with concrete recovery signals can bank a procedure; failed/gave-up runs and clean first-pass successes bank none. Templates cover rejected proposal repair, red-oracle repair, reviewer rejection, and bounded escalation. Each skill records category, exact workflow, trigger tokens, confidence, occurrence and verified-success counts, source session IDs, use count, applied-session IDs, timestamps, and last use. Repeated verified recoveries strengthen the same language/category skill ID rather than duplicating it. Retrieval scores current goal/task tokens plus deterministic blocker categories, selects at most three, deduplicates use accounting within one session, and injects a bounded `procedural-skills` prompt section with confidence and verified-run provenance. The model cannot author or mark a skill successful.

Causal proof establishes four negative/positive boundaries: failed recovery banks zero skills; clean success invents none; repeated verified TypeScript proposal recovery strengthens one stable procedure; unrelated CSS explanation retrieves none. Product-loop proof loads a prior registry from disk, injects “Copy the SEARCH block verbatim” into a later matching worker prompt, records application provenance, and opens the registry through the native artifact command. Existing reflection A/B fixtures now prove a real reflection-on terminal success banks a skill while the reflection-off failed lane persists an empty registry. Oracle procedures require actual oracle-reflection evidence rather than any generic reflection.

Compile, smoke, 100/100 worker stress, extension-host E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.67.0`; inspected visual proof shows `skill 1/2` in the existing compact footer. ## Phase 66 - Transactional Multi-File Command Execution (implemented; copy/worktree/conflict/rollback/host/visual gates PASS)

Reconciled gap: Phase 65 isolated explicit file-edit tools, but `run_command` still executed in the active workspace and only recorded a before/after effect ledger. A command could partially modify many files, fail after mutation, overwrite concurrent user changes, or touch shared Git metadata from a detached worktree.

Implemented result: validated `run_command` proposals now execute through `TransactionalCommandExecutor` in a detached temporary worktree for Git roots or a full filtered workspace copy for non-Git roots. Current dirty source files overlay the Git worktree before execution. Forge computes the complete staged created/modified/deleted set, rejects symlinks, caps transactions at 200 changed files and 20 MiB, checks every changed source path against its pre-execution SHA-256/size state before any merge, and applies the set only after all paths pass. A merge failure restores every changed path from byte backups. Failed commands never merge staged effects. Explicit command references to the active workspace root are rejected before worker launch, and command policy now blocks Git index/history/config/worktree mutations while retaining read-only status/diff/log inspection.

`.forge/worker-command-transactions.json` records command, role, staging mode, base commit, changed/created/modified/deleted paths, merged files/bytes, conflict, rollback, cleanup, PID, duration, and error. `.forge/command-effects.json` links each committed effect entry to its transaction ID/mode. Native artifact opening and compact `txn edit+command/conflicts` telemetry exist.

Behavioral proof covers a three-path copy transaction (create+modify+delete), failed command non-merge, explicit source-root rejection before worker launch, concurrent user-edit conflict with zero merged paths, injected second-file merge failure with successful first-file rollback, dirty Git-worktree two-file execution, worktree registry cleanup, shared Git mutation rejection, persisted main-loop linkage, and artifact opening. Compile, smoke, 100/100 worker stress, extension-host E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.66.0`; inspected visual proof shows `txn 1+1/0`.

Honest boundary: this controls cwd-relative command effects and rejects explicit active-root paths, but it is not kernel filesystem containment. A process exploiting an unknown absolute path, OS facility, child process, or unclassified socket remains outside this guarantee. Excluded dependency/build directories are not copied into command staging; commands requiring them should use dedicated verification tools or require a future container/sandbox implementation.

## Phase 65 - Transactional Role File Edits (implemented; worktree/sparse/conflict/host/visual gates PASS)

Reconciled gap: Phase 62 moved validated tools into role-tagged child processes, but `apply_patch` and `write_file` still mutated the active workspace from those processes. A stale proposal could overwrite user edits made between validation and commit, and the role boundary had no isolated filesystem transaction or merge evidence.

Implemented result: validated file mutations now execute through `TransactionalEditExecutor`. Git-backed workspaces create a detached temporary worktree at `HEAD`, overlay the current target bytes so dirty user state is the actual patch base, execute the role-tagged worker there, compare the active target's SHA-256 before merge, and copy back only the validated target when the source hash is unchanged. Non-git workspaces use an explicit sparse staging directory with the same optimistic concurrency check. Temporary worktrees/staging roots are removed before dispatch resolves. `.forge/worker-edit-transactions.json` records transaction ID, role, proposal, target, mode, source/staged hashes, base commit, worker PID, conflict/commit/cleanup outcome, duration, and error; native artifact opening and compact `txn transactions/conflicts` telemetry exist.

Negative proof changes the active target from a fake worker after staging but before merge; Forge rejects the transaction and preserves the concurrent user bytes. Git proof starts from committed baseline plus dirty target bytes, applies a patch that only matches the dirty overlay, merges the result, and proves `git worktree list --porcelain` is identical before and after. Non-git proof commits through sparse staging. Compile, smoke, 100/100 worker lifecycle stress, extension-host E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.65.0`; inspected visual proof shows `txn 1/0`.

Phase-close boundary: Phase 65 covered one target file per `apply_patch`/`write_file` proposal and did not cover commands; Phase 66 above supersedes that command limitation with bounded multi-file staging. Forge still lacks persistent autonomous role worktrees, deterministic cross-worker branch merge, bounded parallel workers, socket containment, and OS/container sandboxing.

## Phase 64 - Opt-In Hybrid Semantic Retrieval (implemented; mocked causal/cache/fallback gates PASS, live uplift pending credential/opt-in)

Reconciled gap: `.forge/retrieval-index.json` ranked files by path/content token overlap, remembered-file boost, and file type. It could not retrieve conceptually related code when the goal and source used different vocabulary, and Phase 8 still lacked an embedding path, vector cache, cosine ranking, or semantic provenance.

Implemented result: `EmbeddingProvider` abstraction plus `OpenRouterEmbeddingProvider` uses the official batched `POST /api/v1/embeddings` contract, default model `openai/text-embedding-3-small`, normalized vectors, cosine similarity, 32-input batches, and content/model/path-addressed `.forge/embedding-cache.json`. External source embedding is explicitly opt-in through `forge.semanticRetrievalEnabled` (default false) with configurable `forge.embeddingModel`; missing credentials, disabled mode, malformed response, or provider errors leave deterministic retrieval active. Before prompt assembly, up to 80 bounded retrievable documents are embedded/cached and semantic scores blend with deterministic ranking. Candidate `source`, cosine score, model/provider/status, cache hits, embedded document count, query, and errors persist in `.forge/semantic-retrieval.json` and retrieval candidates; native artifact opening and compact `sem refreshes/failures` telemetry exist.

Causal proof uses a mock vector provider where “repair login flow” and credential verification share vectors despite no lexical overlap; `src/auth.ts` ranks first and the model prompt contains `semantic cosine` provenance. Repeating identical query/documents makes zero new embedding calls and reports three cache hits. A throwing provider records `status: failed`, increments semantic failures, and retains deterministic candidates without failing the agent run. Validation exposed an existing contamination bug: root `.forge` was not excluded because only `.forge/` descendants matched, and harness-owned root artifacts repeated the goal. Retrieval now excludes the `.forge` directory plus `PLAN.md`, `SCRATCHPAD.md`, `todos.json`, and `evidence_ledger.json`; those artifacts already enter prompts through dedicated state sections. Compile, smoke, worker stress, E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.64.0`; inspected visual proof shows `sem 1/0`. No live source text was sent during this phase.

## Phase 63 - Deterministic Blocker Taxonomy (implemented; local behavioral/visual gates PASS)

Reconciled gap: provider/schema/firewall/review/tool/oracle/budget/no-progress failures were recorded across logs, reflections, AAR counters, and prose halt reasons, but Forge had no durable blocker object or category-specific retry policy. A weak model therefore received generic failure prose and the user could not distinguish active, resolved, or terminal blockers machine-readably.

Implemented result: deterministic `classifyBlocker` maps failures to provider, schema, role capability, workspace scope, command policy, network policy, patch format, patch applicability, generic firewall, pre-commit review, worker process, tool failure, oracle, budget, no-progress, or step-cap categories. `.forge/blockers.json` records source, category, open/resolved/terminal lifecycle, retryability, task/role ownership, occurrence count, timestamps, summary, deterministic suggested action, and resolution. Repeated same-category failures on one role/task update one object; corrected validation/tool/oracle outcomes resolve matching blockers; terminal failure/give-up freezes unresolved blockers as terminal; success closes remaining goal-level blockers with an explicit resolution. Open blockers are required bounded prompt context. Run stats expose event/unresolved/resolved counts; compact footer renders `blk unresolved/events`; terminal blocker counts/categories feed deterministic AAR and lesson banking.

Behavioral proof distinguishes role, network, patch-applicability, and non-retryable budget blockers; proves role/network rejections persist open during bounded reflection; proves cost halt terminalizes a budget blocker; verifies native artifact opening; and confirms successful runs close blockers without erasing occurrence history. While validating this phase, the new worker-process category exposed an actual IPC shutdown race from Phase 62. A failing-first stress script proved `dispatch()` returned while its PID remained alive and locked the fixture. Parent-owned disconnect/exit handling replaced timing and ACK schemes; `npm run test:workers` now proves 100/100 sequential workers exit before dispatch resolves, and final E2E state records zero worker-process failures. Compile, smoke, worker stress, E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.63.0`; inspected visual proof shows `blk 1/3` in the compact footer.

## Phase 62 - Process-Isolated Workspace Tools (implemented; local behavioral/visual gates PASS, container isolation still missing)

Reconciled gap: Phase 61 enforced role tool scopes and provider session identity, but all workspace tools still executed inside the extension-host process. A tool crash, blocked command, or filesystem operation therefore shared the host's process and inherited environment boundary.

Implemented result: after parent-owned PROPOSE and VALIDATE, workspace reads/searches/edits/tests/diffs/commands execute through `ProcessWorkerExecutor` in a short-lived role-tagged child process. The child receives only workspace root, role, and the validated proposal over IPC; its inherited environment is allowlisted before launch, with blocked key names recorded. It returns the structured tool result and PID, then exits. Parent state/oracles/reflection remain authoritative. Worker timeout, startup failure, early exit, and tool failure return failed tool results and enter the existing reflection/halt path. `.forge/worker-contexts.json` now records process executions/failures, last PID/duration, and blocked environment keys per role; run stats expose `workerProcessExecutions`/`workerProcessFailures`, rendered compactly as `proc executions/failures`.

Behavioral proof writes a fixture file through an Editor child whose PID differs from the extension host, strips `FORGE_WORKER_SECRET`, proves a descendant command prints `false` for that key, and propagates an unknown-tool failure without false success. Full harness E2E terminaled success with five process executions across external role PIDs. The first integration run found that upstream-stripped keys vanished from the inner command-sandbox ledger; the fix unions worker-boundary denials into command metadata, preserving complete transitive environment evidence. Compile, smoke, E2E, visual, package, and Antigravity install/list gates pass for `kennyg.forge-agent@0.62.0`; inspected visual proof shows `proc 4/0` in the compact footer. This is real process isolation for tool execution, not a container, kernel sandbox, persistent autonomous worker, or parallel role execution.

## Executive Verdict

Forge Agent is now a real VS Code/Antigravity extension with a basic success-harness shape, not a cloned IDE. It has an installable VSIX, compact agent UI, OpenRouter provider layer, typed tool registry, deterministic firewall, persistent run artifacts, an autonomous run command, and an early weak-model eval.

It is not yet a mature success-driven coding agent by the standard of the full research blueprint. The deterministic loop, repair gradient, reviewer gates, verification fixtures, budgets, artifact ledgers, enforced role capabilities, role-scoped provider sessions, worktree isolation, optional hybrid semantic retrieval, and multi-tier eval harness exist. The remaining hard gaps are repeated durable live evaluation, real-repository use of the worktree path, live semantic-retrieval uplift measurement, true process/worktree-per-role parallel workers, socket-level containment, and OS/container sandboxing.

Blunt status: Forge has reached an early Founding-release harness, but not the Quality or Triple-A research target.

## Research Consensus

All four research passes converge on the same thesis: the harness matters more than the raw model.

- ChatGPT research emphasizes a durable goal loop, six persistent state objects, OpenRouter routing policy, artifact-first trust, and eval discipline.
- Claude research emphasizes a simple while-tool-use loop wrapped in enforced scaffolding: task state, scratchpad memory, verification, context engineering, edit-format reliability, and sub-agent isolation.
- Gemini research emphasizes an autonomous command center: stateful orchestration, local verification, and background managers that coordinate long-running work.
- The consolidated Blueprint converts those ideas into machine-verifiable phases and non-negotiables: all mutation must flow through `PROPOSE -> VALIDATE -> COMMIT -> NARRATE`, and terminal success must be backed by green oracle evidence.

For this project, all "IDE" language should be read as "extension-host integration." We should use VS Code/Antigravity native editors, terminals, file explorer, and diff surfaces instead of rebuilding an IDE.

## Useful Takeaways

- The IDE-fork idea should remain scrapped. Nothing in the useful research requires a standalone IDE; the essential parts map cleanly to an extension-host harness.
- The most valuable research idea is still the firewall: model proposes, deterministic code validates and commits, model narrates after the fact.
- The second most valuable idea is verification-as-carrier. Forge should prioritize failed-test repair loops over more model-picker UI.
- The third most valuable idea is artifact discipline. Plans, todos, scratchpads, and evidence should be automatically generated and enforced, but not permanently visible as clutter.
- The weak-model eval is useful, but only as an instrument. It should not become the product's main purpose.
- "Sub-agents" should not be implemented as UI personas. They need isolated context, narrow tool permissions, and handoff artifacts, or they are not real sub-agents.
- Background managers and parallelism are attractive but should wait until single-run correctness is stronger.

## Lessons Adopted from Anthropic's "Plan Big, Execute Small" Cookbook (2026-07, reconciled against our live swarm data)

- RATE SPLIT IS THE ENGINE: multi-agent pays when the token-heavy leg bills at a cheaper rate than the judgment leg (their team: 84–98% of input at worker rate → 2.5× cheaper, 3× faster at matched rigor). Our 7B-coordinating-7B swarm has no split, which explains its measured 2.5× cost for no headline gain. → Next major experiment: ARCHITECT/EDITOR lane — mid-tier planner + 7B executors, fourth lane in the tier-2 A/B.
- BRIEF GRANULARITY HAS AN OPTIMUM: their "more, narrower briefs raised the bill" independently replicates our tiny-fixture overhead finding (1.08–1.18×). → Size-gate explorer granularity: no worker below a file-size threshold; batch small files into one brief.
- THE STANDARD ONLY COVERS WHAT'S IN IT: their premise error (wrong park list, perfectly verified) shows decomposition needs its own audit. Our held-out judge audits fixes, not plans. → Add a premise/plan verification pass (one cheap delegation or deterministic check) before execution.
- CHEAP READERS SUMMARIZE AWAY JUDGMENT: their third failure case is our seam-blindness, generalized. Deterministic extracts (interlocks) beat smarter summaries for structural facts; raw slices beat summaries when small. Banked as a swarm design law.
- METERING: adopt per-worker cost attribution in the swarm report (explorer vs implementer split), matching their per-thread usage discipline.

## Phase 69 - Structured Clarification Gates

Status: **Implemented and visually validated**.

Forge now closes a major autonomy/safety gap between blind assumption and manual pause. The proposal schema carries confidence and material uncertainty, and exposes `ask_user` as a non-mutating structured tool. Deterministic code rejects any ordinary proposal when the model reports material uncertainty, persists focused questions under `.forge/clarifications.json`, and holds the run in `awaiting_input` without additional provider calls. The normal bottom chat composer records the answer and resumes the same run with the answer in required prompt context.

Boundaries remain deliberate: the agent must investigate repository facts itself; questions are for user-owned intent, scope, authorization, behavior, cost, and acceptance decisions. Requests are capped at three and duplicate questions terminate rather than loop. This is not proof that every model calibrates uncertainty well; it is proof that self-reported uncertainty cannot silently mutate the workspace.

- Do not build or fork a full IDE.
- Do not recreate editor tabs, terminals, file trees, diff viewers, or browser panes inside Forge.
- Do not treat visual agent personas as sub-agents unless they have isolated context and constrained tools.
- Do not claim full blueprint compliance from package/install success.
- Do not count deterministic fallback success as model-driven success.
- Do not expand UI complexity until the harness loop, verification, reflection, and evals are stronger.
