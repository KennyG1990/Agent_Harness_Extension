# Research-to-Implementation Gap Analysis

## Executive Verdict

Forge Agent is now a real VS Code/Antigravity extension with a basic success-harness shape, not a cloned IDE. It has an installable VSIX, compact agent UI, OpenRouter provider layer, typed tool registry, deterministic firewall, persistent run artifacts, an autonomous run command, and an early weak-model eval.

It is not yet a mature success-driven coding agent by the standard of the full research blueprint. The biggest gap is no longer UI or packaging. The biggest gap is harness depth: repair/no-progress/accountability, bounded reflection, a first reviewer diff gate, reviewer critique artifacts, blocking pre-commit review artifacts, verification fixture matrix, bounded escalation routing, first-pass context bundle rehydration, persisted role handoff artifacts, deterministic retrieval candidates, manifest-backed safety checkpoints, command side-effect capture with sanitized command environment metadata, explicit wall-clock/cost budget halts, isolated workspace-copy execution, and a 15-task mocked weak-model eval now exist, but true git worktree execution, OS/container sandboxing, semantic retrieval, and larger live eval coverage are still missing or proof-level.

Blunt status: Forge has reached an early Founding-release harness, but not the Quality or Triple-A research target.

## Research Consensus

All four research passes converge on the same thesis: the harness matters more than the raw model.

- ChatGPT research emphasizes a durable goal loop, six persistent state objects, OpenRouter routing policy, artifact-first trust, and eval discipline.
- Claude research emphasizes a simple while-tool-use loop wrapped in enforced scaffolding: task state, scratchpad memory, verification, context engineering, edit-format reliability, and sub-agent isolation.
- Gemini research emphasizes an autonomous command center: stateful orchestration, local verification, and background managers that coordinate long-running work.
- The consolidated Blueprint converts those ideas into machine-verifiable phases and non-negotiables: all mutation must flow through `PROPOSE -> VALIDATE -> COMMIT -> NARRATE`, and terminal success must be backed by green oracle evidence.

For this project, all "IDE" language should be read as "extension-host integration." We should use VS Code/Antigravity native editors, terminals, file explorer, and diff surfaces instead of rebuilding an IDE.

## Current Implementation Snapshot

Current Forge implementation, based on source and `BUILD_LOG.md` evidence:

- Extension packaging exists, including VSIX packaging and Antigravity disk-level refresh.
- Provider layer exists with OpenRouter, OpenAI-compatible fallback, `session_id`, fallback `models`, JSON-schema response format, catalog refresh, model sorting, and cost/context metadata.
- Tool registry exists for repo search, symbol search, file reads, patch application, commands, tests, diffs, plan/task/evidence updates, and success declaration.
- Firewall exists for schema validation, workspace path containment, command policy checks, patch applicability, manifest-backed safety checkpoints, targeted restore, workspace snapshot restore, and rejected-action behavior.
- Verification oracles exist for tests, typecheck/lint/build-like checks where detectable, and no-test-suite rejection.
- Basic autonomous run loop exists with inspect, plan, edit, verify, evidence phases.
- Normal runs now persist provider calls/failures, schema repair attempts, model-driven proposals, fallback proposals/actions, validation failures, no-progress turns, and `actuallyModelDriven`.
- Firewall rejection, tool failure, and red-oracle failure now queue bounded reflection notes before retrying, and reflection counters persist in normal run state.
- Mutating proposals now pass a pre-commit review step after deterministic firewall validation and before COMMIT. The ledger persists under `.forge/precommit-reviews.json`; configured review models can block a patch before filesystem mutation, otherwise a deterministic approval is recorded.
- Success now requires a persisted reviewer diff gate. `get_diff` records `.forge/diff-reviews.json`, `.forge/reviewer-critiques.json`, reviewer counters, critique counters, and diff excerpts on evidence entries before final success.
- `run_command` now captures before/after workspace file effects and persists `.forge/command-effects.json` with created, modified, and deleted file lists, output excerpt, sanitized command environment metadata, allowed/blocked env key names, timeout, duration, exit code, and signal.
- Runs now carry explicit budget state under `.forge/budget.json`. Wall-clock caps halt before provider/tool work, and cost caps halt after provider usage is recorded but before validation/commit can mutate the workspace.
- Isolated run API exists through `forge-agent.runIsolatedAgentGoal` and Proof-tab controls. It copies the source workspace to a temp root, runs the normal harness against the copy, compares source before/after, and persists `.forge/isolated-runs/latest-isolated-run.json` plus a diff summary.
- Verification fixture matrix now covers passing tests, failing tests, missing tests, typecheck failure, lint failure, malformed patch, out-of-workspace path, and blocked command behavior. The report persists at `.forge/verification-fixture-matrix.json`.
- The verification fixture matrix also includes an unsolvable step-cap case that proves exhausted runs terminate as `gave_up` without green evidence.
- Bounded escalation routing now sends proposals to an escalation/review model binding after repeated reflection failures and persists `.forge/escalations.json`.
- Context engineering now persists `.forge/context-bundle.json` with goal, open tasks, recent files, reflection/escalation/review notes, scratchpad summary, retrieval policy, token estimate, and compaction flag. The bundle is fed into the next proposal prompt.
- Deterministic retrieval now ranks likely relevant workspace files from goal/task text, persists `.forge/retrieval-index.json`, stores `retrievalCandidates` in the context bundle, and feeds those candidates into proposal prompts.
- Role handoff artifacts now persist under `.forge/role-handoffs.json` with active role, scoped allowed tools, responsibilities, role-open tasks, recent context, and a handoff summary. The next proposal prompt consumes only the active role's handoff slice.
- Safety checkpoints now persist under `.forge/safety-checkpoints.json`; each mutating proposal records checkpoint strategy, protected paths, manifest path, and compact `safe` run status. Targeted file checkpoints restore original file content in tests.
- Persistent artifacts exist: `.forge/state.json`, goal contract, task graph, plan, evidence ledger, repository knowledge, skill registry, plus `PLAN.md`, `SCRATCHPAD.md`, `todos.json`, and evidence files.
- Weak-model eval exists as a 15-task disposable fixture loop with mocked default path and opt-in live direction. The latest verified mocked scorecard records bare solved 1/15, harness solved 15/15, actually model-driven 15, and fallback solved 0.
- Compact UI exists and avoids cloned IDE surfaces.

Important limitation: the harness exists, but it is not yet mature. Current agency is basic and still needs deeper self-repair, richer state transitions, real worker isolation, worktree execution, process sandboxing, and larger evals.

## Useful Takeaways

- The IDE-fork idea should remain scrapped. Nothing in the useful research requires a standalone IDE; the essential parts map cleanly to an extension-host harness.
- The most valuable research idea is still the firewall: model proposes, deterministic code validates and commits, model narrates after the fact.
- The second most valuable idea is verification-as-carrier. Forge should prioritize failed-test repair loops over more model-picker UI.
- The third most valuable idea is artifact discipline. Plans, todos, scratchpads, and evidence should be automatically generated and enforced, but not permanently visible as clutter.
- The weak-model eval is useful, but only as an instrument. It should not become the product's main purpose.
- "Sub-agents" should not be implemented as UI personas. They need isolated context, narrow tool permissions, and handoff artifacts, or they are not real sub-agents.
- Background managers and parallelism are attractive but should wait until single-run correctness is stronger.

## Phase 57 — TIER-4: TASKS THAT CANNOT BE PATTERN-MATCHED (specced after live tier-3 saturation)

CONFESSION DRIVING THE DESIGN: tier-3 goals name the defective symbol, so planning reduces to goal→file regex mapping — the same-model architect's live 5/5 proves focus works, NOT that reasoning works. Tier-4's one law: **no goal may contain any symbol, file name, or string that appears at the defect site.** Enforce with a generation-time assertion (the banked truncation-horizon technique): tokenize the goal, assert zero overlap with identifiers within N lines of the seeded defect.

Task dimensions (each raises a different wall):
1. SYMPTOM-ONLY GOALS: "discounted items total wrong at checkout" — behavioral description; the planner must trace data flow to locate the cause. This is where regex-planners and reasoning-planners finally separate (re-run the DeepSeek-vs-7B architect A/B HERE).
2. CAUSAL CHAINS: fixes requiring 2-3 coordinated edits where any single edit alone leaves tests red (the true choke point; suspect-rotation and single-targetFile plans both break — plans need multi-step structure, PLAN_SCHEMA gains ordered subtasks).
3. DISTRACTORS: seeded plausible-but-innocent anomalies (weird-but-correct code, stale comments claiming bugs, near-miss function names) that pattern-matchers bite on; judge unchanged — biting a distractor costs steps and shows in the metrics.
4. SPEC-GAP FEATURES: "implement X per this behavioral spec" where the held-out judge tests edge cases the goal does NOT enumerate — measures design judgment; workspace-test authoring becomes load-bearing.
5. REAL REPOS (merge tier-3 slice 2 here): clone into Phase-35 worktrees, real bugs or reverted fixes, held-out judge = the repo's own withheld test. The final wall.
Metrics to add: located-correct-file-by-step-N (finding speed), distractor-bite rate, plan-subtask coherence. ## Lessons Adopted from Anthropic's "Plan Big, Execute Small" Cookbook (2026-07, reconciled against our live swarm data)

- RATE SPLIT IS THE ENGINE: multi-agent pays when the token-heavy leg bills at a cheaper rate than the judgment leg (their team: 84–98% of input at worker rate → 2.5× cheaper, 3× faster at matched rigor). Our 7B-coordinating-7B swarm has no split, which explains its measured 2.5× cost for no headline gain. → Next major experiment: ARCHITECT/EDITOR lane — mid-tier planner + 7B executors, fourth lane in the tier-2 A/B.
- BRIEF GRANULARITY HAS AN OPTIMUM: their "more, narrower briefs raised the bill" independently replicates our tiny-fixture overhead finding (1.08–1.18×). → Size-gate explorer granularity: no worker below a file-size threshold; batch small files into one brief.
- THE STANDARD ONLY COVERS WHAT'S IN IT: their premise error (wrong park list, perfectly verified) shows decomposition needs its own audit. Our held-out judge audits fixes, not plans. → Add a premise/plan verification pass (one cheap delegation or deterministic check) before execution.
- CHEAP READERS SUMMARIZE AWAY JUDGMENT: their third failure case is our seam-blindness, generalized. Deterministic extracts (interlocks) beat smarter summaries for structural facts; raw slices beat summaries when small. Banked as a swarm design law.
- METERING: adopt per-worker cost attribution in the swarm report (explorer vs implementer split), matching their per-thread usage discipline.

## Non-Goals / Scrapped IDE Assumptions

- Do not build or fork a full IDE.
- Do not recreate editor tabs, terminals, file trees, diff viewers, or browser panes inside Forge.
- Do not treat visual agent personas as sub-agents unless they have isolated context and constrained tools.
- Do not claim full blueprint compliance from package/install success.
- Do not count deterministic fallback success as model-driven success.
- Do not expand UI complexity until the harness loop, verification, reflection, and evals are stronger.
